# 向量資料庫與 OpenAI Embeddings 技術指南

本文檔旨在為非技術背景的使用者，解釋本系統核心技術：向量資料庫與 OpenAI Embeddings 的運作原理。

---

## 1. 傳統資料庫 vs. 向量資料庫

### 傳統資料庫 (SQL)
傳統資料庫就像一個**Excel 表格**，存儲的是精確的數據（數字、文字、日期）。
- **搜尋方式**：關鍵字精確匹配。
- **例子**：搜尋 "蘋果"，只會找到包含 "蘋果" 的資料，找不到 "Apple" 或 "水果"。
- **優點**：精確、結構化。
- **缺點**：不懂語意，無法理解 "相似" 的概念。

### 向量資料庫 (Vector DB)
向量資料庫就像一個**多維座標系**，存儲的是資料的「意義」（向量）。
- **搜尋方式**：計算距離（相似度）。
- **例子**：搜尋 "蘋果"，系統知道它跟 "Apple"、"水果"、"iPhone" 很像，跟 "卡車" 很不像。
- **優點**：理解語意、模糊搜尋、跨語言。
- **本系統應用**：當您問 "機台亮紅燈怎麼辦？" 時，系統能找到 "N706 異常警報處理流程" 的文件，即使文件中沒有 "亮紅燈" 這三個字（可能是寫 "警示燈號異常"）。

---

## 2. 什麼是 Embeddings (向量)?

Embedding 是將文字、圖片或聲音轉換為一串數字（向量）的技術。這串數字代表了該內容的「語意特徵」。

### 簡單的比喻
想像我們用兩個數字來描述食物：[甜度, 熱量]。
- 🍎 蘋果：[0.7, 0.4]
- 🍰 蛋糕：[0.9, 0.9]
- 🥦 花椰菜：[0.1, 0.1]

在這個座標系中，蘋果和蛋糕在 "甜度" 上比較近；蘋果和花椰菜在 "熱量" 上比較近。

### OpenAI text-embedding-3-small
本系統使用 OpenAI 的 `text-embedding-3-small` 模型。
- 它不是用 2 個數字，而是用 **1536 個數字** 來描述一段文字！
- 這意味著它能捕捉極其細微的語意差異（如情感、時態、專業術語關聯、因果關係等）。

## 3. RAG (檢索增強生成) 如何運作？

本系統結合了向量搜尋與 GPT 生成能力，流程如下：

1. **使用者提問**： "如何保養 N706 機台？"
2. **向量化**：將問題轉換為 1536 維的向量。
3. **向量搜尋**：在資料庫中尋找與該向量 "距離最近" 的文件切片（最相似的內容）。
4. **生成回答**：將找到的文件內容作為 "參考資料"，連同使用者的問題一起交給 GPT-4。
5. **AI 回答**：GPT-4 閱讀參考資料後，用自然語言回答您的問題。

---

## 4. 為什麼有時候搜尋不準？

雖然向量搜尋很強大，但也有局限：

1. **關鍵字太獨特**：如特定的錯誤代碼 "E-12345"，如果向量沒抓到這個特徵，傳統關鍵字搜尋反而更準（所以本系統 v4.0 採用混合搜尋）。
2. **切片不完整**：如果文件被切得太碎，上下文遺失，AI 可能看不懂。
3. **語意模糊**：如 "那個東西壞了"，AI 不知道 "那個東西" 是什麼。

### 如何提升準確度？
- **提供完整上下文**：在文件中把主詞寫清楚（"N706 機台" 優於 "它"）。
- **使用專業術語**：讓 Embeddings 能捕捉到精確的領域特徵。
- **結構化撰寫**：參考 `FILE_FORMATS.md` 的建議格式。
