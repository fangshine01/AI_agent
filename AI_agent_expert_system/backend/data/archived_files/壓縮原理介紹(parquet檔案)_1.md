# 資料壓縮原理與演算法深度指南 (Deep Dive into Data Compression)

這份文件詳細介紹大數據與資料庫底層使用的壓縮原理，特別針對 Parquet 等欄式儲存格式中常用的技術進行剖析。

## 1. 壓縮的核心概念

*   **無失真壓縮 (Lossless)**: 壓縮後還原的資料與原始資料完全一致 (如 Zip, Gzip, Snappy)。資料庫和程式碼必須使用此類。
*   **失真壓縮 (Lossy)**: 允許遺失部分細節以換取極高壓縮率 (如 JPEG, MP3)。資料分析通常**不使用**此類 (除非是特定的科學計算近似值)。

## 2. 智慧編碼技術 (Smart Encodings) - 輕量級壓縮

在真正進入 Gzip/Snappy 等重量級演算法之前，Parquet 會先使用「編碼技術」來減少資料的 Bit 寬度。這通常是效果最顯著的步驟。

### 2.1 差分編碼 (Delta Encoding)

**原理**: 對於數值是**遞增**、**遞減**或**變化很小**的序列，儲存「當前值與前一值的差 (Delta)」會比儲存原始值更省空間。

*   **適用場景**: 時間戳記 (Timestamp)、自動遞增 ID (Auto-increment ID)、有序的計數器。

#### **詳細範例**

假設我們有一組 Unix 時間戳記 (秒)，代表連續的事件：

**原始資料 (Int32, 每個佔 4 Bytes)**:
```text
1600000000, 1600000001, 1600000005, 1600000010, 1600000012
|__________||__________||__________||__________||__________|
  4 Bytes     4 Bytes     4 Bytes     4 Bytes     4 Bytes    = 20 Bytes
```

**Parquet Delta Encoding 處理步驟**:

1.  **Block 1 (Header)**: 儲存起始值 `1600000000` (佔 4 Bytes)。
2.  **Block 2 (Deltas)**: 計算與前一個數的差值。
    *   `1600000001 - 1600000000 = +1`
    *   `1600000005 - 1600000001 = +4`
    *   `1600000010 - 1600000005 = +5`
    *   `1600000012 - 1600000010 = +2`
3.  **儲存結果**: `1, 4, 5, 2`。

**再結合 Bit Packing (位元封裝)**:
這些差值 (1, 4, 5, 2) 最大的數是 5，只需要 **3 bits** ($2^3=8$) 就能存下所有差值，不需要用完整的 32 bits。

```text
起始值: 1600000000 (4 Bytes)
差值: 
  1 -> 001
  4 -> 100
  5 -> 101
  2 -> 010
總共: 001100101010 (12 bits = 1.5 Bytes)

總大小: 4 + 1.5 = 5.5 Bytes (原始為 20 Bytes，壓縮率約 72.5%)
```

> **重點**: 數值越大，差分編碼的效果通常越好，因為它把「大整數」變成了「小整數」。

---

### 2.2 遊程編碼 (Run-Length Encoding, RLE)

**原理**: 當資料出現**連續重複**的值時，只儲存「值」和「重複次數」。

*   **適用場景**: 分類資料 (Category)、狀態碼 (Status Code)、排序過的資料。

#### **詳細範例**

**原始資料**:
`"A", "A", "A", "A", "B", "B", "C", "A", "A"`

**RLE 編碼後**:
`"A":4, "B":2, "C":1, "A":2`

> **Parquet 中的 RLE**: Parquet 實際上使用的是 **RLE + Bit-Packing 混合模式**。它會智慧地判斷這段資料適合 RLE (有很多重複) 還是適合 Bit-Packing (隨機但數值小)。

---

### 2.3 字典編碼 (Dictionary Encoding)

**原理**: 將重複出現的**字串**映射為短小的**整數 (ID)**。

*   **適用場景**: 國家代碼、產品類別、使用者性別、瀏覽器類型。**這是 Parquet 預設且最強大的編碼之一**。

#### **詳細範例**

**原始資料 (String)**:
`"IPhone", "Samsung", "IPhone", "Pixel", "Samsung", "IPhone"`

**步驟**:
1.  **建立字典**:
    *   `0: "IPhone"`
    *   `1: "Samsung"`
    *   `2: "Pixel"`
2.  **轉換資料**:
    `0, 1, 0, 2, 1, 0`
3.  **封裝**:
    現在我們只需要存 0~2 的整數。這只需要 **2 bits** ($2^2=4$)。原始字串 "IPhone" 佔 6 Bytes (48 bits)，現在只要 2 bits，壓縮率極高。

---

### 2.4 位元封裝 (Bit Packing)

**原理**: 不使用標準的 Byte (8 bits) 對齊，而是緊湊地將 bit 塞在一起。

*   **適用場景**: 所有數值範圍有限的整數。

#### **詳細範例**

假設我們有一組數值: `0, 1, 2, 3, 4, 5, 6, 7`。
每個數都在 0~7 之間，只需要 **3 bits**。

**標準儲存 (Int8)**:
`00000000`, `00000001`, ... (每個都浪費前面 5 個 0)

**Bit Packing**:
將它們緊密排列: `00000101 00111001 ...`

---

### 2.5 Frame of Reference (FOR) 編碼

**原理**: 如果資料不是遞增的 (無法用 Delta)，但都在某個範圍內 (例如體重 50~100 KG)，可以選一個「基準值 (Reference)」，只存「偏移量 (Offset)」。

**原始資料**:
`85, 90, 88, 92, 80` (假設這是考試成績)

**Reference = 80**:
偏移量: `+5, +10, +8, +12, +0`

這些偏移量很小，只需 4 bits 就能存下 (0~15)，遠比原始的 Int 需要更少空間。

---

## 3. 通用壓縮演算法 (Block Compression)

上述編碼完成後，資料已經變小很多。Parquet 還會對整個 Data Page 再做一次通用的二進位壓縮。

### 3.1 LZ77 / LZ78 與滑動視窗 (Sliding Window)

這是 Gzip, Snappy, LZ4, Zstd 的核心基礎。

**原理**:
*   讀取器維護一個「視窗 (Window)」，看過的資料還在記憶體中。
*   當遇到新資料時，回頭看「這段資料以前出現過嗎？」
*   如果出現過，就用 `(距離 distance, 長度 length)` 來取代。

#### **詳細範例**

**原始字串**:
`"blablablabla"`

**壓縮過程**:
1.  先存 `"bla"`。
2.  接著又遇到 `"bla"`。
3.  壓縮器發現 3 個字元前出現過一樣的 `"bla"`。
4.  輸出 `(3, 3)` (向後回溯 3 個字元，複製 3 個字元)。
5.  後面又是 `"bla"`...

**結果**: `"bla" (3, 3) (3, 3)` ... 重複性越高的文字，壓縮率越高。

### 3.2 霍夫曼編碼 (Huffman Coding)

**原理**: 出現頻率**高**的符號，用**短**的編碼；出現頻率**低**的符號，用**長**的編碼。

**範例**:
假設文章中 `e` 出現 1000 次，`z` 出現 1 次。
*   `e` 編碼為 `0` (1 bit)
*   `z` 編碼為 `100101` (6 bits)

這樣整體總長度會最小化。(Deflate 演算法 = LZ77 + Huffman)。

## 4. 總結：Parquet 的組合拳

當你寫入一個 Parquet 檔案時，實際發生了什麼？

1.  **資料**: `[100, 101, 102, 103, 1000]`
2.  **Step 1: Delta Encoding**: `100 (Base), +1, +1, +1, +897`
3.  **Step 2: Bit Packing**: 把 +1 壓成 1 bit，+897 壓成 10 bits。
4.  **Step 3: Block Compression (Snappy/Zstd)**: 把上述處理完的二進位資料塊，再用 LZ77 找重複模式壓縮。

這就是為什麼 Parquet 能夠比 CSV 小數十倍，且掃描速度快數十倍的原因。
